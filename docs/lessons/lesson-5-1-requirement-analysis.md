# Lesson 5-1：需求拆解技巧

## 學習目標

- 掌握需求拆解的方法論
- 能夠將業務需求轉化為技術需求
- 理解銀行系統需求的特殊性

---

## 需求拆解的重要性

在銀行 IT 專案中：
- 業務人員使用業務語言
- 開發人員需要技術規格
- BA 是兩者之間的橋樑

**好的需求拆解 = 成功專案的一半**

---

## 需求拆解步驟

### 1. 需求收集

```
來源：
- 業務單位會議
- 法規要求
- 系統優化建議
- 客戶反饋
```

### 2. 需求分析

```
分析維度：
- 業務流程
- 資料流向
- 系統介面
- 非功能需求
```

### 3. 需求文件化

```
文件類型：
- 需求規格書（BRD/SRD）
- 功能規格書（FSD）
- 介面規格書
- 測試案例
```

## 需求拆解框架

### 5W2H 方法

| 問題 | 說明 | 銀行範例 |
|------|------|----------|
| What | 需求是什麼？ | 新增手機銀行轉帳功能 |
| Why | 為什麼需要？ | 提升客戶便利性，減少分行業務 |
| Who | 使用者是誰？ | 個人網銀客戶 |
| When | 何時上線？ | 2026年Q2 |
| Where | 在哪裡使用？ | 手機 APP |
| How | 如何實現？ | 線上交易 + 後台審核 |
| How much | 成本/效益？ | 開發成本 vs 預期效益 |

## 銀行需求拆解實例

### 案例：新增「快速轉帳」功能

#### 業務需求（原始）

> 「客戶希望能快速轉帳給常用收款人，不用每次都輸入帳號」

#### 拆解過程

**Step 1: 功能拆解**

| 子功能 | 說明 |
|--------|------|
| 收款人管理 | 新增/修改/刪除常用收款人 |
| 快速轉帳入口 | 從收款人列表選擇 |
| 轉帳驗證 | 驗證金額、密碼 |
| 交易執行 | 執行轉帳交易 |

**Step 2: 資料拆解**

| 資料實體 | 欄位 | 來源 |
|----------|------|------|
| 常用收款人 | 客戶編號、收款人帳號、收款人姓名、銀行代碼 | 新增 |
| 交易記錄 | 原有交易欄位 + 收款人類型 | 擴充 |

**Step 3: 系統拆解**

| 系統 | 影響 | 說明 |
|------|------|------|
| 手機 APP | 新功能 | 新增收款人管理、快速轉帳入口 |
| 核心系統 | 介面 | 接收轉帳交易 |
| 後台管理 | 新功能 | 收款人資料維護 |

**Step 4: 流程拆解**

```
新增收款人流程：
1. 客戶輸入收款人資訊
2. 系統驗證銀行代碼、帳號格式
3. 儲存收款人資料
4. 發送通知

快速轉帳流程：
1. 客戶選擇收款人
2. 輸入金額
3. 驗證密碼/OTP
4. 執行轉帳
5. 更新餘額
6. 發送通知
```

## 需求規格書模板

### 1. 需求概述

```markdown
## 需求編號：REQ-2026-001
## 需求名稱：快速轉帳功能
## 需求類型：新功能
## 優先級：高
## 業務單位：零售銀行部
## 預計上線：2026-06-01
```

### 2. 業務背景

```markdown
## 現況問題
客戶每次轉帳都需要輸入完整帳號，操作繁瑣

## 業務目標
提升轉帳便利性，減少客戶操作時間

## 預期效益
- 客戶滿意度提升 10%
- 每筆轉帳時間減少 30 秒
```

### 3. 功能需求

```markdown
## FR-001 收款人管理
客戶可新增最多 20 組常用收款人

## FR-002 快速轉帳
從收款人列表選擇，只需輸入金額即可轉帳

## FR-003 交易限額
單筆限額 50,000，每日累計限額 200,000
```

### 4. 非功能需求

```markdown
## NFR-001 效能
轉帳交易回應時間 < 3 秒

## NFR-002 安全性
需通過 OTP 驗證

## NFR-003 可用性
系統可用率 99.9%
```

## BA 實務技巧

### 需求訪談技巧

| 技巧 | 說明 |
|------|------|
| 開放式問題 | 「請描述目前的操作流程？」 |
| 追問細節 | 「如果金額超過限額怎麼辦？」 |
| 確認理解 | 「所以您的意思是...對嗎？」 |
| 記錄承諾 | 「這個我記下來，會寫進規格書」 |

### 常見需求陷阱

| 陷阱 | 說明 | 解決方式 |
|------|------|----------|
| 需求模糊 | 「系統要快」 | 定義具體指標 |
| 需求變更 | 「再加上...」 | 變更控制流程 |
| 需求遺漏 | 忘記例外處理 | 檢查清單 |
| 需求衝突 | 不同單位意見不同 | 協調會議 |
