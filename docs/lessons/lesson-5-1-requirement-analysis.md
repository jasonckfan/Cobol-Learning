# Lesson 5-1：需求拆解技巧

## 學習目標

- 掌握需求拆解的方法論與實務技巧
- 能夠將業務需求轉化為技術需求
- 理解銀行系統需求的特殊性與合規要求
- 能夠撰寫清晰的需求規格書

---

## 需求拆解的重要性

### BA 的核心角色

在銀行 IT 專案中，BA 是業務與技術之間的橋樑：

```
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│  業務單位   │  ────>  │     BA      │  ────>  │  開發團隊   │
│             │         │             │         │             │
│ 「客戶想要  │         │ 需求分析    │         │ 技術規格    │
│  快速轉帳」 │         │ 拆解轉化    │         │ 程式實作    │
└─────────────┘         └─────────────┘         └─────────────┘
      │                        │                       │
      │    業務語言            │    技術語言           │
      ▼                        ▼                       ▼
  「方便好用」           「功能、流程、資料」      「程式、檔案、介面」
```

### 需求拆解不當的後果

| 問題 | 後果 | 案例 |
|------|------|------|
| 需求遺漏 | 系統功能不完整 | 忘記處理跨幣別轉帳 |
| 需求模糊 | 開發方向錯誤 | 「系統要快」沒有定義具體指標 |
| 需求衝突 | 返工、延遲 | A 部門與 B 部門意見不一致 |
| 需求膨脹 | 專案超時超支 | 不斷新增「順便加上」的功能 |

## 需求拆解步驟

### 1. 需求收集

```
需求來源：
├── 內部來源
│   ├── 業務單位會議
│   ├── 高層策略決策
│   ├── 系統優化建議
│   └── 內部稽核發現
│
├── 外部來源
│   ├── 法規要求（金管會、央行）
│   ├── 客戶反饋與投訴
│   ├── 市場競爭壓力
│   └── 合作夥伴要求
│
└── 技術來源
    ├── 系統維護需求
    ├── 技術債償還
    └── 安全漏洞修補
```

### 2. 需求分析

```
分析維度：
├── 業務流程分析
│   ├── 現行流程（As-Is）
│   ├── 目標流程（To-Be）
│   └── 差異分析（Gap Analysis）
│
├── 資料流向分析
│   ├── 資料來源
│   ├── 資料處理
│   └── 資料輸出
│
├── 系統介面分析
│   ├── 內部介面
│   ├── 外部介面
│   └── 跨系統互動
│
└── 非功能需求分析
    ├── 效能要求
    ├── 安全要求
    └── 合規要求
```

### 3. 需求文件化

```
文件類型：
├── BRD（Business Requirement Document）
│   └── 業務需求書：描述業務目標與需求
│
├── SRD（System Requirement Document）
│   └── 系統需求書：描述系統功能需求
│
├── FSD（Functional Specification Document）
│   └── 功能規格書：詳細功能規格
│
├── Interface Spec
│   └── 介面規格書：系統間介面定義
│
└── Test Case
    └── 測試案例：驗收測試依據
```

## 需求拆解框架

### 5W2H 方法

| 問題 | 說明 | 銀行範例 |
|------|------|----------|
| What | 需求是什麼？ | 新增手機銀行快速轉帳功能 |
| Why | 為什麼需要？ | 提升客戶便利性，減少分行業務量 |
| Who | 使用者是誰？ | 個人網銀客戶、企業網銀客戶 |
| When | 何時上線？ | 2026年Q2，配合行銷活動 |
| Where | 在哪裡使用？ | 手機 APP、網銀 |
| How | 如何實現？ | 線上交易 + 後台審核 + 對帳機制 |
| How much | 成本/效益？ | 開發成本評估 vs 預期效益 |

### INVEST 原則

好的需求應符合 INVEST 原則：

| 原則 | 說明 | 銀行範例 |
|------|------|----------|
| Independent | 獨立性 | 每個功能需求可獨立開發 |
| Negotiable | 可協商 | 需求細節可與開發團隊討論 |
| Valuable | 有價值 | 對業務有實質價值 |
| Estimable | 可估算 | 開發團隊能估算工作量 |
| Small | 足夠小 | 需求範圍適當，不過於龐大 |
| Testable | 可測試 | 有明確的驗收標準 |

## 銀行需求拆解實例

### 案例 1：新增「快速轉帳」功能

#### 業務需求（原始）

> 「客戶希望能快速轉帳給常用收款人，不用每次都輸入帳號」

#### Step 1: 功能拆解

```
功能層次圖：

快速轉帳功能
├── 收款人管理
│   ├── 新增收款人
│   ├── 修改收款人
│   ├── 刪除收款人
│   └── 查詢收款人列表
│
├── 快速轉帳
│   ├── 選擇收款人
│   ├── 輸入金額
│   ├── 交易驗證
│   └── 執行轉帳
│
└── 交易管理
    ├── 查詢交易記錄
    └── 交易通知
```

#### Step 2: 資料拆解

```cobol
      * 新增資料結構：常用收款人檔
       01 FAVORITE-PAYEE.
          05 FP-SEQ-NO          PIC 9(8).      *> 序號
          05 FP-CUST-ID         PIC X(10).     *> 客戶編號
          05 FP-PAYEE-ACCT      PIC X(16).     *> 收款人帳號
          05 FP-PAYEE-NAME      PIC X(40).     *> 收款人姓名
          05 FP-BANK-CODE       PIC X(3).      *> 銀行代碼
          05 FP-BANK-NAME       PIC X(20).     *> 銀行名稱
          05 FP-NICKNAME        PIC X(20).     *> 暱稱
          05 FP-CREATE-DATE     PIC 9(8).      *> 建立日期
          05 FP-LAST-USE-DATE   PIC 9(8).      *> 最後使用日期
          05 FP-STATUS          PIC X.         *> 狀態
          05 FP-TRANS-COUNT     PIC 9(5).      *> 交易次數
```

#### Step 3: 系統拆解

| 系統 | 影響類型 | 說明 |
|------|----------|------|
| 手機 APP | 新功能 | 收款人管理、快速轉帳入口 |
| 網銀系統 | 新功能 | 同手機 APP |
| 核心系統 | 介面 | 接收轉帳交易 |
| 後台管理 | 新功能 | 收款人資料維護、異常監控 |
| 通知系統 | 擴充 | 交易通知增加收款人資訊 |

#### Step 4: 流程拆解

```
新增收款人流程：
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 輸入收款│────>│ 驗證銀行│────>│ 驗證帳號│────>│ 儲存收款│
│ 人資訊  │     │ 代碼    │     │ 格式    │     │ 人資料  │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
                                                      │
                                                      ▼
                                                ┌─────────┐
                                                │ 發送通知│
                                                └─────────┘

快速轉帳流程：
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 選擇收款│────>│ 輸入金額│────>│ 驗證密碼│────>│ 執行轉帳│
│ 人      │     │         │    │ /OTP    │     │         │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
                                                      │
                    ┌─────────────────────────────────┘
                    ▼
              ┌─────────┐     ┌─────────┐
              │ 更新餘額│────>│ 發送通知│
              └─────────┘     └─────────┘
```

### 案例 2：利率調整批次需求

#### 業務需求

> 「配合央行升息，需調整存款利率，並追溯計算利息差額」

#### Step 1: 拆解需求

```
需求拆解：
├── 利率設定
│   ├── 定義新利率
│   ├── 設定生效日期
│   └── 利率版本管理
│
├── 利息追溯計算
│   ├── 識別受影響帳戶
│   ├── 計算利息差額
│   └── 產生調整交易
│
└── 通知與報表
    ├── 客戶通知
    ├── 內部報表
    └── 稽核記錄
```

#### Step 2: 技術需求

```cobol
      * 利率變更記錄檔
       01 RATE-CHANGE-LOG.
          05 RC-SEQ-NO          PIC 9(10).     *> 序號
          05 RC-RATE-TYPE       PIC X(2).      *> 利率類型
          05 RC-OLD-RATE        PIC V9(6).     *> 原利率
          05 RC-NEW-RATE        PIC V9(6).     *> 新利率
          05 RC-EFF-DATE        PIC 9(8).      *> 生效日期
          05 RC-CHANGE-REASON   PIC X(50).     *> 變更原因
          05 RC-APPROVER        PIC X(10).     *> 核准人
          05 RC-APPROVE-TIME    PIC 9(14).     *> 核准時間

      * 利息調整記錄檔
       01 INT-ADJUST-LOG.
          05 IA-SEQ-NO          PIC 9(10).     *> 序號
          05 IA-ACCT-NO         PIC X(16).     *> 帳號
          05 IA-RATE-CHANGE-ID  PIC 9(10).     *> 利率變更編號
          05 IA-ADJUST-AMT      PIC S9(11)V99. *> 調整金額
          05 IA-PERIOD-START    PIC 9(8).      *> 調整期間起
          05 IA-PERIOD-END      PIC 9(8).      *> 調整期間訖
          05 IA-POST-DATE       PIC 9(8).      *> 入帳日期
          05 IA-STATUS          PIC X.         *> 狀態
```

#### Step 3: 批次處理流程

```
利率調整批次流程：

Step 1: 讀取利率變更參數
        ↓
Step 2: 掃描受影響帳戶
        ↓
Step 3: 計算利息差額
        │
        ├── 每日利息差額 = 本金 × (新利率 - 舊利率) / 365
        │
        └── 累計差額 = Σ 每日利息差額
        ↓
Step 4: 產生調整交易
        ↓
Step 5: 入帳處理
        ↓
Step 6: 產生報表與通知
```

## 需求規格書模板

### 完整需求規格書結構

```markdown
# 需求規格書

## 文件控制
| 版本 | 日期 | 作者 | 變更說明 |
|------|------|------|----------|
| 1.0 | 2026/04/06 | BA Team | 初版 |

## 1. 需求概述
### 1.1 需求編號：REQ-2026-001
### 1.2 需求名稱：快速轉帳功能
### 1.3 需求類型：新功能
### 1.4 優先級：高
### 1.5 業務單位：零售銀行部
### 1.6 預計上線：2026-06-01

## 2. 業務背景
### 2.1 現況問題
客戶每次轉帳都需要輸入完整帳號，操作繁瑣

### 2.2 業務目標
提升轉帳便利性，減少客戶操作時間

### 2.3 預期效益
- 客戶滿意度提升 10%
- 每筆轉帳時間減少 30 秒
- 分行臨櫃轉帳業務減少 15%

## 3. 功能需求

### FR-001 收款人管理
客戶可新增最多 20 組常用收款人
- 輸入欄位：收款人帳號、姓名、銀行代碼、暱稱
- 驗證規則：帳號格式、銀行代碼有效性
- 儲存後發送通知

### FR-002 快速轉帳
從收款人列表選擇，只需輸入金額即可轉帳
- 單筆限額：50,000
- 每日累計限額：200,000
- 需要 OTP 驗證

### FR-003 交易記錄查詢
可查詢最近 90 天的交易記錄
- 顯示欄位：日期、收款人、金額、狀態

## 4. 非功能需求

### NFR-001 效能
- 轉帳交易回應時間 < 3 秒
- 收款人列表載入 < 1 秒

### NFR-002 安全性
- 需要 OTP 驗證
- 收款人資料加密儲存
- 異常交易監控

### NFR-003 可用性
- 系統可用率 99.9%
- 維護時段外 24x7 服務

### NFR-004 合規性
- 符合個資法規定
- 交易記錄保存 5 年
- 符合金管會相關規定

## 5. 介面需求

### 5.1 手機 APP 介面
- 收款人列表頁面
- 新增/編輯收款人頁面
- 快速轉帳頁面

### 5.2 核心系統介面
- 轉帳交易介面（現有）
- 收款人資料儲存介面（新增）

## 6. 資料需求
### 6.1 新增資料表
- FAVORITE_PAYEE（常用收款人）

### 6.2 異動資料表
- TRANS_LOG（交易明細，新增欄位）

## 7. 業務規則
### BR-001 收款人數量限制
每位客戶最多 20 組收款人

### BR-002 轉帳限額規則
- 單筆：50,000
- 每日：200,000
- 超過限額需臨櫃申請

## 8. 驗收標準
- 完成所有功能測試案例
- 通過安全測試
- 效能測試達標
- UAT 測試通過

## 9. 風險與假設
### 9.1 風險
- 效能可能受網路影響
- 外部銀行系統可能不穩定

### 9.2 假設
- 核心系統介面不需修改
- 客戶願意使用新功能

## 10. 附錄
- 流程圖
- 畫面線稿圖
- 資料字典
```

## BA 實務技巧

### 需求訪談技巧

| 技巧 | 說明 | 範例 |
|------|------|------|
| 開放式問題 | 引導對方詳細說明 | 「請描述目前的操作流程？」 |
| 追問細節 | 挖掘隱藏需求 | 「如果金額超過限額怎麼辦？」 |
| 確認理解 | 避免誤解 | 「所以您的意思是...對嗎？」 |
| 記錄承諾 | 留下書面記錄 | 「這個我記下來，會寫進規格書」 |
| 期待管理 | 不輕易承諾 | 「這個需求我需要和技術團隊確認可行性」 |

### 需求訪談檢核清單

```
訪談前準備：
□ 了解相關業務背景
□ 準備訪談問題清單
□ 確認參與人員
□ 預約會議室/視訊

訪談中記錄：
□ 記錄關鍵需求
□ 記錄決策事項
□ 記錄待確認事項
□ 記錄風險與假設

訪談後跟進：
□ 發送會議紀錄
□ 確認需求理解正確
□ 追蹤待確認事項
□ 更新需求文件
```

### 常見需求陷阱

| 陷阱 | 說明 | 解決方式 |
|------|------|----------|
| 需求模糊 | 「系統要快」、「要好用」 | 追問具體指標：「多快算快？」 |
| 需求變更 | 「再加上...」 | 建立變更控制流程 |
| 需求遺漏 | 忘記例外處理 | 使用檢查清單 |
| 需求衝突 | 不同單位意見不同 | 召開協調會議，記錄決策 |
| 需求膨脹 | 不斷新增功能 | 定義範圍邊界，嚴格控制變更 |
| 隱含假設 | 以為「大家都知道」 | 明確記錄所有假設 |

## 銀行特殊考量

### 合規要求

```
常見監管要求：
├── 金管會
│   ├── 銀行法
│   ├── 個人資料保護法
│   └── 金融消費者保護法
│
├── 央行
│   ├── 外匯管理規定
│   └── 大額交易申報
│
├── 國際標準
│   ├── PCI DSS（支付卡）
│   ├── SOX（內控）
│   └── Basel（資本適足）
│
└── 內部規範
    ├── 資安政策
    ├── 作業規範
    └── 稽核要求
```

### 安全性需求

| 領域 | 要求 | 說明 |
|------|------|------|
| 認證 | 多因子認證 | 密碼 + OTP + 生物特徵 |
| 授權 | 最小權限原則 | 只給必要的權限 |
| 加密 | 資料加密 | 傳輸加密、儲存加密 |
| 稽核 | 交易記錄 | 所有操作可追蹤 |
| 監控 | 異常偵測 | 即時監控異常交易 |
