## 為什麼測試案例很重要？

### 銀行系統測試的特殊性

```
銀行系統特點：
├── 涉及金錢
│   ├── 錯誤可能導致財務損失
│   ├── 利息計算必須精確
│   └── 餘額必須隨時正確
│
├── 涉及客戶資料
│   ├── 個資保護要求
│   ├── 資料安全要求
│   └── 法規遵循要求
│
├── 24x7 營運
│   ├── 系統可用性要求高
│   ├── 災難復原要求
│   └── 效能要求嚴格
│
└── 複雜的業務規則
    ├── 多種產品組合
    ├── 多種計算方式
    └── 多種例外處理
```

### 測試不當的後果

| 案例 | 問題 | 後果 |
|------|------|------|
| 利息計算錯誤 | 測試未覆蓋閏年 | 客戶投訴、監管關注 |
| 餘額更新失敗 | 未測試並發交易 | 帳務不平、客戶損失 |
| 限額檢核遺漏 | 只測正向案例 | 詐騙交易、監管罰款 |
| 個資外洩 | 未做安全測試 | 違反個資法、商譽受損 |

## 測試案例設計方法

### 1. 等價類別分析（Equivalence Partitioning）

將輸入分成有效和無效等價類別，每個類別只需測試一個代表值。

```
範例：轉帳金額輸入

有效等價類別：
├── 1 - 30,000（一般限額內）
├── 30,001 - 50,000（需 OTP）
└── 50,001 - 200,000（需額外驗證）

無效等價類別：
├── 0 或負數
├── 非數字字元
├── 超過每日限額 200,000
└── 小數位數超過 2 位
```

| 輸入欄位 | 有效等價類別 | 無效等價類別 |
|----------|--------------|--------------|
| 帳號 | 16 碼數字 | 空白、少於 16 碼、含字母、不存在 |
| 金額 | 1 - 50,000 | 0、負數、超過限額、非數字 |
| 幣別 | USD, HKD, CNY | 空白、不支援的幣別、小寫 |
| 日期 | 有效日期格式 | 無效日期、未來日期、太久以前 |

### 2. 邊界值分析（Boundary Value Analysis）

測試邊界條件，因為錯誤最常發生在邊界。

```
範例：單筆轉帳限額 = 50,000

測試案例：
├── 49,999（邊界內）→ 應成功
├── 50,000（邊界值）→ 應成功
├── 50,001（邊界外）→ 應失敗
├── 1（最小值）→ 應成功
└── 0（最小值-1）→ 應失敗
```

```
範例：每日累計限額 = 200,000

測試案例：
├── 第 1 筆：50,000 → 成功，累計 50,000
├── 第 2 筆：50,000 → 成功，累計 100,000
├── 第 3 筆：50,000 → 成功，累計 150,000
├── 第 4 筆：49,999 → 成功，累計 199,999
├── 第 5 筆：1 → 成功，累計 200,000
├── 第 6 筆：1 → 失敗，超過每日限額
```

### 3. 決策表（Decision Table）

處理複雜條件組合。

```
範例：轉帳交易授權規則

條件：
├── C1: 帳戶狀態 = 正常
├── C2: 餘額充足
├── C3: 金額 <= 單筆限額
├── C4: 今日累計 + 金額 <= 每日限額
└── C5: OTP 驗證通過

決策表：
┌──────┬───┬───┬───┬───┬───┬───┬───┬───┐
│ 規則 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │ 8 │
├──────┼───┼───┼───┼───┼───┼───┼───┼───┤
│ C1   │ Y │ Y │ Y │ Y │ N │ Y │ Y │ Y │
│ C2   │ Y │ Y │ Y │ N │ - │ Y │ Y │ Y │
│ C3   │ Y │ Y │ N │ - │ - │ Y │ Y │ Y │
│ C4   │ Y │ N │ - │ - │ - │ Y │ Y │ Y │
│ C5   │ Y │ - │ - │ - │ - │ N │ Y │ Y │
├──────┼───┼───┼───┼───┼───┼───┼───┼───┤
│ 結果 │成功│超限│超額│餘額│帳戶│OTP │成功│成功│
│      │   │   │   │不足│異常│失敗│   │   │
└──────┴───┴───┴───┴───┴───┴───┴───┴───┘
```

### 4. 狀態轉換測試（State Transition Testing）

測試系統狀態轉換。

```
範例：貸款申請狀態

狀態圖：
┌─────────┐   提交申請   ┌─────────┐
│  初始   │ ─────────> │ 審核中  │
└─────────┘             └────┬────┘
                             │
              ┌──────────────┼──────────────┐
              ▼              ▼              ▼
        ┌─────────┐   ┌─────────┐   ┌─────────┐
        │ 核准    │   │ 退件    │   │ 拒絕    │
        └────┬────┘   └────┬────┘   └─────────┘
             │              │
             ▼              ▼
        ┌─────────┐   ┌─────────┐
        │ 撥款    │   │ 補件    │
        └────┬────┘   └────┬────┘
             │              │
             ▼              └──────> ┌─────────┐
        ┌─────────┐                 │ 審核中  │
        │ 結案    │                 └─────────┘
        └─────────┘

測試案例：
├── 正常流程：初始 → 審核中 → 核准 → 撥款 → 結案
├── 退件補件：初始 → 審核中 → 退件 → 補件 → 審核中 → 核准
├── 直接拒絕：初始 → 審核中 → 拒絕
└── 無效轉換：核准 → 拒絕（不應發生）
```

## 銀行測試案例分類

### 1. 功能測試

```
功能測試分類：

正向測試（Happy Path）：
├── 正常流程測試
├── 各種合法輸入組合
└── 標準業務場景

負向測試（Negative Testing）：
├── 輸入驗證錯誤
├── 業務規則違反
├── 系統錯誤處理
└── 例外情況處理

邊界測試（Boundary Testing）：
├── 最小值
├── 最大值
├── 邊界值±1
└── 空值/空白
```

### 2. 非功能測試

| 類型 | 說明 | 測試重點 |
|------|------|----------|
| 效能測試 | 回應時間、吞吐量 | 交易 < 3 秒、1000 TPS |
| 負載測試 | 系統負載能力 | 高峰期處理能力 |
| 壓力測試 | 系統極限 | 超過預期負載的反應 |
| 安全測試 | 安全性驗證 | SQL 注入、XSS、權限 |
| 相容性測試 | 環境相容 | 瀏覽器、作業系統 |
| 可靠性測試 | 穩定性 | 長時間運行、故障恢復 |

### 3. 回歸測試

```
回歸測試目的：
├── 確認變更未影響現有功能
├── 驗證系統整體穩定性
└── 確保沒有引入新問題

回歸測試策略：
├── 完整回歸：測試所有功能
├── 部分回歸：測試受影響功能
└── 風險導向：依風險決定測試範圍
```

## BA 實務技巧

### 測試案例設計原則

| 原則 | 說明 | 範例 |
|------|------|------|
| 完整性 | 覆蓋所有功能和場景 | 包含正向、負向、邊界 |
| 獨立性 | 每個測試案例可獨立執行 | 不依賴其他測試結果 |
| 可重複 | 可重複執行，結果一致 | 使用固定測試數據 |
| 可追蹤 | 可追溯到需求規格 | 標註對應需求編號 |
| 清晰性 | 步驟清楚，預期明確 | 避免模糊描述 |

### 測試數據準備

```
測試數據類型：
├── 帳戶數據
│   ├── 正常帳戶（各種狀態）
│   ├── 凍結帳戶
│   ├── 結清帳戶
│   └── 透支帳戶
│
├── 交易數據
│   ├── 各金額範圍
│   ├── 各幣別
│   └── 各交易類型
│
├── 時間數據
│   ├── 平常日
│   ├── 月底/年底
│   ├── 假日
│   └── 閏年
│
└── 極端數據
    ├── 最大金額
    ├── 最小金額
    └── 特殊字元
```

### 測試覆蓋率檢核

```
功能覆蓋率檢核：
□ 所有功能需求都有對應測試案例
□ 所有業務規則都有對應測試案例
□ 所有介面欄位都有輸入驗證測試
□ 所有錯誤訊息都有觸發測試

路徑覆蓋率檢核：
□ 所有分支邏輯都有測試
□ 所有迴圈都有測試
□ 所有例外處理都有測試

資料覆蓋率檢核：
□ 各種資料狀態都有測試
□ 各種資料組合都有測試
□ 邊界值都有測試
```
