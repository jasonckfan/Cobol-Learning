### 1.2 CLM 相關的主要監管報表

| 報表類型 | 法規依據 | 提交頻率 | CLM 相關性 |
|----------|----------|----------|------------|
| **LCR (流動性覆蓋比率)** | Basel III | 月報 | 高 — 資金池餘額影響 HQLA |
| **NSFR (淨穩定資金比率)** | Basel III | 季報 | 高 — 存款穩定性評估 |
| **大額曝險報表** | 本地法規 | 月報 | 中 — 單一客戶存款上限監控 |
| **存款統計報表** | 央行要求 | 月報/季報 | 高 — 各類存款產品餘額 |
| **外幣敞口報表** | 外匯管理 | 月報 | 高 — MCN 多貨幣風險 |
| **利率風險報表** | IRRBB | 季報 | 中 — 資金池利率敏感度 |

## 二、LCR 報表深度解析

### 2.1 什麼是 LCR？

**LCR (Liquidity Coverage Ratio)** = HQLA / 淨現金流出 ≥ 100%

- **HQLA (High Quality Liquid Assets)**: 高流動性資產
- **淨現金流出**: 30 天壓力情境下的預期資金流出

```
┌─────────────────────────────────────────────────────────────────┐
│              LCR 計算與 CLM 的關聯                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   HQLA (分子)                                    CLM 影響       │
│   ─────────────                                  ─────────      │
│   • 現金及央行準備金                              直接相關      │
│   • Level 1 資產 (主權債等)                       間接相關      │
│   • Level 2A/2B 資產                              間接相關      │
│                                                                 │
│   淨現金流出 (分母)                              CLM 影響       │
│   ─────────────────                              ─────────      │
│   • 零售存款流出                                  ⭐高度相關    │
│   • 批發存款流出                                  ⭐高度相關    │
│   • 同業拆借流出                                  中度相關      │
│                                                                 │
│   CLM 資金池餘額會影響：                                         │
│   • 存款分類 (穩定 vs 較不穩定)                                  │
│   • 流出比率假設 (run-off factor)                               │
│   • 最終的淨現金流出計算                                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.4 存款穩定性評分邏輯

```cobol
       *=============================================================
       * 存款穩定性評分計算
       *=============================================================
       3000-CALCULATE-STABILITY-SCORE.
           MOVE ZERO TO WS-STABILITY-SCORE
           
           *---- 基礎分數 (依客戶類型) -----------------------------
           EVALUATE LCR-CUSTOMER-TYPE
               WHEN 'R'  * 零售存款
                   MOVE 90 TO WS-BASE-SCORE
               WHEN 'S'  * 中小企業
                   MOVE 80 TO WS-BASE-SCORE
               WHEN 'W'  * 批發客戶
                   MOVE 50 TO WS-BASE-SCORE
               WHEN 'F'  * 金融機構
                   MOVE 20 TO WS-BASE-SCORE
               WHEN OTHER
                   MOVE 50 TO WS-BASE-SCORE
           END-EVALUATE
           
           *---- 調整因子 -------------------------------------------
           * 帳戶年齡調整 (老客戶更穩定)
           IF WS-ACCOUNT-AGE-YEARS > 5
               ADD 5 TO WS-STABILITY-ADJUSTMENT
           ELSE IF WS-ACCOUNT-AGE-YEARS > 2
               ADD 2 TO WS-STABILITY-ADJUSTMENT
           END-IF
           
           * 餘額波動調整 (波動小更穩定)
           IF WS-BALANCE-VOLATILITY < 10
               ADD 3 TO WS-STABILITY-ADJUSTMENT
           ELSE IF WS-BALANCE-VOLATILITY > 50
               SUBTRACT 10 FROM WS-STABILITY-ADJUSTMENT
           END-IF
           
           * 關聯交易調整 (操作帳戶較穩定)
           IF LCR-OPERATIONAL-FLAG = 'Y'
               ADD 5 TO WS-STABILITY-ADJUSTMENT
           END-IF
           
           *---- 計算最終分數 ---------------------------------------
           COMPUTE LCR-STABILITY-SCORE = 
               WS-BASE-SCORE + WS-STABILITY-ADJUSTMENT
           
           * 確保分數在 0-100 範圍內
           IF LCR-STABILITY-SCORE > 100
               MOVE 100 TO LCR-STABILITY-SCORE
           ELSE IF LCR-STABILITY-SCORE < 0
               MOVE 0 TO LCR-STABILITY-SCORE
           END-IF
           .

       *=============================================================
       * 流出係數指派
       *=============================================================
       4000-ASSIGN-RUN-OFF-FACTOR.
           *---- 依據 Basel III 標準指派流出係數 --------------------
           
           EVALUATE TRUE
               * 零售存款 - 穩定
               WHEN LCR-RETAIL-DEPOSIT AND 
                    LCR-STABILITY-SCORE >= 80
                   MOVE 0.03 TO LCR-RUN-OFF-FACTOR
               
               * 零售存款 - 較不穩定
               WHEN LCR-RETAIL-DEPOSIT AND 
                    LCR-STABILITY-SCORE < 80
                   MOVE 0.05 TO LCR-RUN-OFF-FACTOR
               
               * 中小企業存款
               WHEN LCR-SMALL-BUSINESS
                   MOVE 0.10 TO LCR-RUN-OFF-FACTOR
               
               * 批發存款 - 有擔保
               WHEN LCR-WHOLESALE AND 
                    LCR-INSURED-FLAG = 'Y'
                   MOVE 0.20 TO LCR-RUN-OFF-FACTOR
               
               * 批發存款 - 無擔保
               WHEN LCR-WHOLESALE AND 
                    LCR-INSURED-FLAG = 'N'
                   MOVE 0.40 TO LCR-RUN-OFF-FACTOR
               
               * 金融機構存款
               WHEN LCR-FINANCIAL
                   MOVE 1.00 TO LCR-RUN-OFF-FACTOR
               
               WHEN OTHER
                   MOVE 1.00 TO LCR-RUN-OFF-FACTOR
           END-EVALUATE
           
           *---- 計算加權流出金額 -----------------------------------
           COMPUTE LCR-WEIGHTED-OUTFLOW = 
               LCR-BALANCE-AMT * LCR-RUN-OFF-FACTOR
           .

```

## 三、報表驗證與對帳機制

### 3.1 為什麼驗證如此重要？

```
┌─────────────────────────────────────────────────────────────────┐
│              監管報表錯誤的嚴重性                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  監管報表錯誤可能導致：                                          │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 法規處罰                                              │   │
│  │     • 罰款 (可達數百萬美元)                               │   │
│  │     • 業務限制                                            │   │
│  │     • 監管評級下降                                        │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  2. 聲譽風險                                              │   │
│  │     • 媒體負面報導                                        │   │
│  │     • 投資者信心下降                                      │   │
│  │     • 客戶流失                                            │   │
│  ├─────────────────────────────────────────────────────────┤   │
│  │  3. 營運風險                                              │   │
│  │     • 需重新提交更正報表                                  │   │
│  │     • 額外稽核成本                                        │   │
│  │     • 系統緊急修復                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  因此，多層驗證機制是必須的！                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 三層驗證架構

```
┌─────────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────────┤

│  ┌─────────────────────────────────────────────────────────┐   │

│  │  • 必填欄位檢查                                          │   │

│  │  • 異常值警示 (如：單一客戶餘額超過閾值)                 │   │

│                              ↓                                  │
│  ┌─────────────────────────────────────────────────────────┐   │

│  │  ─────────────────────────────────────────────────       │   │
│  │  • 與源系統對帳 (CLM 總餘額 = 報表彙總)                  │   │
│  │  • 產品分類正確性抽查                                    │   │
│  │  • 客戶類型標記驗證                                      │   │
│  │  • 計算公式驗證 (手工抽樣計算)                           │   │
│  │  • 跨報表一致性檢查 (LCR vs 存款統計報表)                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Layer 3: 法規合規驗證 (Regulatory Compliance)           │   │
│  │  ─────────────────────────────────────────────           │   │
│  │  • 法規部門審查報表樣本                                  │   │
│  │  • 與歷史趨勢比對分析                                    │   │
│  │  • 同業比較 (如可取得)                                   │   │
│  │  • 最終簽核 (CFO / 風控長)                               │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│                        提交監管機構                             │
│                                                                 │

```

## 五、常見問題與解答

### Q1: 監管報表錯誤如何更正？

**A:** 
更正流程：
1. **立即通知**: 告知法規部門與管理層
2. **影響評估**: 評估錯誤對 LCR 比率的影響程度
3. **根因分析**: 找出錯誤來源 (資料/邏輯/人為)
4. **更正報表**: 產生更正後的報表
5. **提交說明**: 向監管機構提交錯誤說明與更正
6. **流程改進**: 更新驗證機制防止再次發生

**BA 注意**: 建立清晰的異常追蹤表，記錄每個錯誤的處理過程。

### Q2: 如何處理法規變更？

**A:** 
法規變更處理步驟：
1. **影響評估**: 分析變更對現有報表的影響範圍
2. **需求規格**: 撰寫變更需求規格書
3. **開發測試**: 實施變更並進行完整測試
4. **平行測試**: 新舊邏輯並行驗證
5. **過渡期安排**: 確認生效日期與過渡安排
6. **文件更新**: 更新所有相關文件

**BA 注意**: 與法規部門保持密切溝通，確保理解變更意圖。

### Q3: 跨系統資料不一致怎麼辦？

**A:** 
常見原因與處理：
- **時間差**: 不同系統的 cutoff time 不同 → 統一 cutoff 時間
- **定義差異**: 同一欄位在不同系統定義不同 → 建立對照文件
- **匯率差異**: 使用不同匯率來源 → 統一匯率來源
- **科目對應**: 會計科目對應錯誤 → 更新對照表

**BA 注意**: 建立定期的跨系統對帳機制，及早發現不一致。

### Q4: 如何證明報表計算正確？

**A:** 
建議的證據留存：
1. **計算邏輯文件**: 詳細的公式與規則說明
2. **測試案例**: 完整的測試覆蓋證明
3. **對帳記錄**: 每次報表的對帳清單
4. **簽核文件**: 各級主管的審查簽名
5. **系統日誌**: 批次執行的完整日誌
6. **變更記錄**: 所有邏輯變更的審批記錄

## 六、練習題

### 練習 1: LCR 計算驗證

給定以下資料，請計算 LCR 比率：

**HQLA (高流動性資產)**:
- Level 1 資產：$ 500,000,000
- Level 2A 資產：$ 200,000,000 (折價係數 0.85)
- Level 2B 資產：$ 100,000,000 (折價係數 0.50)

**存款資料**:
| 客戶類型 | 餘額 | 流出係數 |
|----------|------|----------|
| 零售穩定 | $ 300,000,000 | 0.03 |
| 零售較不穩定 | $ 200,000,000 | 0.05 |
| 批發有擔保 | $ 400,000,000 | 0.20 |
| 批發無擔保 | $ 300,000,000 | 0.40 |

**問題**:
1. 計算總 HQLA (記得應用折價係數)
2. 計算總加權現金流出
3. 計算 LCR 比率
4. 判斷是否符合監管要求 (≥ 100%)

### 練習 2: 需求分析

某法規更新要求：
> 「零售存款的穩定性評分需納入『交易頻率』因子：過去 12 個月平均每月交易 > 10 筆的，穩定性分數扣 10 分」

**問題**:
1. 這個需求對 CLM 系統有什麼影響？
2. 需要從哪個系統取得交易頻率資料？
3. 如何設計測試案例驗證這個新邏輯？
4. 歷史報表是否需要追溯調整？

### 練習 3: 異常處理設計

情境：LCR 報表產生時，發現某零售客戶的餘額較上月暴增 500%。

**問題**:
1. 系統應如何處理這個異常？ (繼續計算/標記警示/暫停報表)
2. 需要通知哪些相關人員？
3. 如何調查這個異常的原因？
4. 如果確認是正常業務 (如大額存款)，如何記錄與追蹤？
