# Lesson 5-6：UAT 測試規劃

## 學習目標

- 理解 UAT 的目的與流程

## 什麼是 UAT？

### 定義

UAT（User Acceptance Testing，使用者驗收測試）是由業務使用者執行的測試，目的是驗證系統是否符合業務需求。

```
測試階層：

單元測試 (Unit Test)
    │ 由開發人員執行
    │ 測試程式碼正確性
    ▼
整合測試 (Integration Test)
    │ 由 QA 執行
    │ 測試模組間整合
    ▼
系統測試 (System Test)
    │ 由 QA 執行
    │ 測試完整功能
    ▼
使用者驗收測試 (UAT)  ← 本課程重點
    │ 由業務使用者執行
    │ 驗證業務需求
    ▼
上線
```

### UAT 與其他測試的差異

| 項目 | 系統測試 (SIT) | 使用者驗收測試 (UAT) |
|------|----------------|----------------------|
| 執行者 | QA 人員 | 業務使用者 |
| 目的 | 驗證功能正確性 | 驗證符合業務需求 |
| 測試案例 | 技術導向 | 業務場景導向 |
| 測試數據 | 模擬數據 | 模擬真實業務數據 |
| 驗收標準 | 功能規格 | 業務需求 |

### 為什麼 UAT 很重要？

```
UAT 的重要性：
├── 業務把關
│   └── 確保系統符合業務實際需求
│
├── 最後防線
│   └── 上線前發現問題的最後機會
│
├── 使用者培訓
│   └── 讓業務人員提前熟悉系統
│
└── 正式驗收
    └── 作為系統交付的正式依據
```

### 各階段詳細說明

```
1. 計畫階段（約 1-2 週）
   ├── 確定測試範圍
   │   └── 哪些功能需要測試？
   │
   ├── 制定測試策略
   │   └── 測試重點、優先順序
   │
   ├── 規劃時程
   │   └── 測試開始/結束日期
   │
   ├── 安排資源
   │   └── 測試人員、環境、工具
   │
   └── 風險評估
       └── 可能影響測試的因素

2. 準備階段（約 1-2 週）
   ├── 設計測試案例
   │   └── 以業務場景為基礎
   │
   ├── 準備測試數據
   │   └── 模擬真實業務數據
   │
   ├── 建置測試環境
   │   └── UAT 專用環境
   │
   └── 測試人員培訓
       └── 說明測試方法與流程

3. 執行階段（約 1-2 週）
   ├── 執行測試案例
   │   └── 依據測試計畫執行
   │
   ├── 記錄測試結果
   │   └── 通過/失敗/阻塞
   │
   └── 回報缺陷
       └── 發現問題立即記錄

4. 驗收階段（約 1-2 天）
   ├── 缺陷清單確認
   ├── 業務單位簽核
   └── 測試報告完成
```

## 測試案例設計

### UAT 測試案例特點

```
UAT 測試案例 vs 系統測試案例：

系統測試案例：
├── 以功能規格為基礎
├── 涵蓋所有功能點
├── 包含技術細節
└── 由 QA 設計

UAT 測試案例：
├── 以業務場景為基礎
├── 涵蓋核心業務流程
├── 使用業務語言
└── 由 BA 協助業務設計
```

### 業務場景導向設計

```
場景設計步驟：

Step 1: 識別業務場景
        「客戶臨櫃辦理存款」

Step 2: 分析場景步驟
        1. 客戶到達櫃台
        2. 出示身分證件
        3. 告知存款金額
        4. 櫃員輸入交易
        5. 確認交易成功
        6. 列印憑條

Step 3: 設計測試案例
        - 正常流程
        - 例外流程（證件過期、金額異常等）

Step 4: 準備測試數據
        - 測試帳號
        - 測試金額
        - 測試日期
```

## 測試數據準備

### 測試數據分類

```
測試數據類型：

1. 帳戶數據
   ├── 正常帳戶（各種狀態）
   ├── 凍結帳戶
   ├── 結清帳戶
   ├── 透支帳戶
   └── 新開戶

2. 交易數據
   ├── 各金額範圍
   ├── 各幣別
   ├── 各交易類型
   └── 歷史交易記錄

3. 參數數據
   ├── 利率設定
   ├── 限額設定
   ├── 幣別設定
   └── 營業日設定

4. 特殊數據
   ├── 月底/年底數據
   ├── 假日數據
   ├── 大額交易
   └── 邊界值數據
```

### 測試數據準備流程

```
測試數據準備步驟：

Step 1: 盤點需求
        - 哪些測試場景需要什麼數據？
        - 需要多少筆測試數據？

Step 2: 設計數據結構
        - 帳戶資料
        - 交易資料
        - 參數資料

Step 3: 產生測試數據
        - 手動建立
        - 或從生產環境脫敏複製
        - 或使用數據產生工具

Step 4: 驗證數據
        - 確認數據正確性
        - 確認數據完整性

Step 5: 匯入測試環境
        - 執行數據匯入
        - 驗證匯入結果
```

## 缺陷追蹤管理

### 缺陷嚴重程度分級

| 等級 | 定義 | 範例 | 處理時效 |
|------|------|------|----------|
| Critical | 系統無法使用或核心功能失效 | 系統當機、資料遺失 | 立即處理 |
| Major | 主要功能無法正常使用 | 存款失敗、餘額錯誤 | 24 小時內 |
| Minor | 次要功能異常或不影響主要流程 | 報表格式錯誤 | 48 小時內 |
| Enhancement | 改善建議 | 畫面調整、文案修改 | 上線後處理 |

### 缺陷狀態流程

```
缺陷狀態流程：

│ (新增)  │     │ (指派)  │     │ (已修正)│
└─────────┘     └─────────┘     └────┬────┘
                                     │
                                     ▼
┌─────────┐     ┌─────────┐     ┌─────────┐
│ Closed  │<────│Verified │<────│ Retest  │
│ (關閉)  │     │(已驗證) │     │ (重測)  │

     │

```

## BA 實務技巧

### UAT 準備檢核清單

```
測試前準備：
□ 測試計畫已核准
□ 測試案例已設計完成
□ 測試數據已準備完成
□ 測試環境已建置完成
□ 測試人員已安排
□ 測試工具已準備
□ 啟動會議已完成

測試中監控：
□ 每日測試進度追蹤
□ 缺陷及時回報
□ 缺陷修復進度追蹤
□ 風險即時通報
□ 測試日誌記錄

測試後驗收：
□ 所有測試案例已執行
□ Critical/Major 缺陷已修復
□ Minor 缺陷已有處理計畫
□ 測試報告已完成
□ 業務單位已簽核
```

### 測試完成準則

```
UAT 完成準則：

功能性：
□ 所有高優先級測試案例通過率 100%
□ 所有測試案例通過率 >= 95%
□ Critical 缺陷數 = 0
□ Major 缺陷數 = 0（或有變通方案）

非功能性：
□ 效能測試通過
□ 安全測試通過
□ 相容性測試通過

文件：
□ 測試報告完成
□ 缺陷清單確認
□ 已知問題記錄

驗收：
□ 業務單位簽核
□ 測試單位簽核
□ 專案經理簽核
```

### 常見問題處理

| 問題 | 處理方式 |
|------|----------|
| 測試環境不穩定 | 聯繫維運團隊，必要時延後測試 |
| 測試數據不足 | 緊急準備測試數據 |
| 測試人員不足 | 協調其他資源或延長測試時間 |
| 缺陷修復延遲 | 協調開發優先級，必要時延後上線 |
| 需求理解差異 | 召開協調會議釐清 |
