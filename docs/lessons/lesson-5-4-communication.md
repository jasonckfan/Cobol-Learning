# Lesson 5-4：與開發人員溝通技巧

## 學習目標

- 掌握與開發人員溝通的要領
- 能夠有效傳達需求與業務邏輯
- 建立良好的協作關係
- 能夠處理溝通衝突與困難

---

## 為什麼溝通很重要？

### 溝通不良的代價

```
溝通不良的後果：
├── 需求理解錯誤
│   └── 開發方向偏差 → 返工、延遲
│
├── 技術限制未溝通
│   └── 無法實現的需求 → 重新設計
│
├── 資訊傳遞不完整
│   └── 遺漏功能 → 缺陷、客戶投訴
│
└── 優先級認知不同
    └── 資源配置衝突 → 專案延遲
```

## BA 與開發的語言差異

### 業務語言 vs 技術語言

| BA 說的 | 開發聽到的 | 應該怎麼說 |
|---------|------------|------------|
| 「系統要快」 | 「要多快？」 | 「回應時間 < 3 秒，95% 交易 < 2 秒」 |
| 「要安全」 | 「安全有很多種」 | 「需要 OTP 驗證 + Session 15 分鐘過期」 |
| 「操作簡單」 | 「簡單的定義是？」 | 「三步驟內完成，不需說明文件」 |
| 「資料要準確」 | 「本來就應該準確」 | 「四捨五入到小數點第二位」 |
| 「不能出錯」 | 「怎麼定義出錯？」 | 「交易成功率 > 99.9%，失敗有明確錯誤訊息」 |
| 「介面要好看」 | 「好看是主觀的」 | 「符合 UI 設計規規範，參考附件設計稿」 |

### 常見術語對照

| 業務術語 | 技術術語 | 說明 |
|----------|----------|------|
| 客戶 | Customer Entity | 客戶資料實體 |
| 帳戶 | Account Record | 帳戶記錄 |
| 交易 | Transaction | 交易處理 |
| 餘額 | Balance Field | 餘額欄位 |
| 對帳 | Reconciliation | 對帳處理程序 |
| 批次 | Batch Job | 批次作業 |
| 報表 | Report | 報表輸出 |
| 介面 | Interface/API | 系統介面 |

## 有效溝通技巧

### 1. 具體化（Be Specific）

```
❌ 模糊：
「客戶體驗要好」

✅ 具體：
「頁面載入時間 < 2 秒」
「操作步驟 < 3 步」
「錯誤訊息要清楚，包含解決方法」
「支援手機瀏覽器（iOS Safari, Android Chrome）」
```

### 2. 結構化（Structure）

```markdown
## 功能需求：快速轉帳

### 業務背景
客戶每次轉帳需輸入完整帳號，操作繁瑣，降低使用意願。

### 功能描述
客戶可預設常用收款人，轉帳時從列表選擇，減少輸入。

### 業務規則
1. 每位客戶最多設定 20 組收款人
2. 單筆轉帳限額：50,000 元
3. 每日累計限額：200,000 元
4. 超過 30,000 元需 OTP 驗證

### 輸入欄位
| 欄位 | 類型 | 長度 | 必填 | 驗證規則 |
|------|------|------|------|----------|
| 收款人帳號 | 文字 | 16 | 是 | 數字格式 |
| 收款人姓名 | 文字 | 40 | 是 | - |
| 銀行代碼 | 文字 | 3 | 是 | 銀行代碼檢核 |

### 輸出結果
- 成功：顯示成功訊息，更新餘額，發送通知
- 失敗：顯示錯誤原因，提供解決建議

### 例外處理
| 情境 | 處理方式 |
|------|----------|
| 餘額不足 | 顯示錯誤：「餘額不足，目前餘額 X 元」 |
| 超過限額 | 顯示錯誤：「超過單筆限額，限額 X 元」 |
| 帳號不存在 | 顯示錯誤：「收款人帳號不存在」 |
```

### 4. 範例化（Example）

```
提供具體範例：

範例 1：正常轉帳
- 輸入：帳號 1234567890123456，金額 10,000
- 預期：成功，餘額從 50,000 變為 40,000

範例 2：超過限額
- 輸入：帳號 1234567890123456，金額 60,000
- 預期：失敗，顯示「超過單筆限額 50,000 元」

範例 3：餘額不足
- 輸入：帳號 1234567890123456，金額 10,000
- 帳戶餘額：5,000
- 預期：失敗，顯示「餘額不足，目前餘額 5,000 元」
```

## 常見溝通場景

### 場景 1：需求說明會

```
會議前準備：
├── 發送會議邀請
│   ├── 主旨：[需求說明] 快速轉帳功能
│   ├── 參與者：開發 Team Lead、架構師、QA Lead
│   └── 時間：預留 1-2 小時
│
├── 準備會議資料
│   ├── 需求規格書（PDF）
│   ├── 流程圖（Visio/Draw.io）
│   ├── 畫面線稿（Figma/Sketch）
│   └── 問題清單（預先整理）
│
└── 設備檢查
    ├── 投影設備
    └── 視訊會議軟體

會議進行：
├── 開場（5 分鐘）
│   └── 說明會議目的、議程
│
├── 需求說明（30 分鐘）
│   ├── 業務背景
│   ├── 功能概述
│   ├── 詳細功能說明
│   └── 業務規則
│
├── 問題討論（20 分鐘）
│   ├── 技術可行性
│   ├── 效能考量
│   └── 時程評估
│
└── 總結（5 分鐘）
    ├── 行動項目指派
    └── 後續會議安排

會議後跟進：
├── 發送會議紀會議紀錄
├── 更新需求文件
├── 追蹤待辦事項
└── 安排後續會議
```

```
BA 角度：
├── 說明業務目的
│   └── 「這個功能是為了讓客戶...」

├── 提供業務情境

│
├── 說明業務規則
│   └── 「根據金管會規定，必須...」
│

│   └── 「現有架構限制，無法...」

└── 提出風險警示

```

## 處理溝通困難

### 常見溝通障礙

| 障礙 | 原因 | 解決方式 |
|------|------|----------|
| 術語不同 | 專業領域差異 | 建立詞彙表、多用範例 |
| 假設不一致 | 隱含假設未說明 | 明確列出所有假設 |
| 優先級不同 | 立場不同 | 協調會議、管理層決策 |
| 時間壓力 | 專案時程緊 | 精簡溝通、抓重點 |
| 地理分散 | 遠距工作 | 善用視訊、文件化 |

### 衝突處理

```
衝突類型：

1. 需求衝突
   - 業務單位要求 vs 技術限制
   - 處理方式：協調會議、找出雙贏方案

2. 時程衝突
   - 業務期望時程 vs 開發評估時程
   - 處理方式：討論優先級、分階段交付

3. 品質衝突
   - 完整測試 vs 時間壓力
   - 處理方式：風險評估、決策記錄

溝通原則：
├── 保持專業態度
├── 以事實為基基礎
├── 尋求共識
├── 必要時升級處理
└── 記錄決策過程
```

## 文件化最佳實務

### 規格書撰寫要點

| 要點 | 說明 | 範例 |
|------|------|------|
| 清楚 | 一句話一個意思 | 「單筆限額 50,000 元」 |
| 完整 | 包含所有必要資訊 | 「含正常處理和例外處理」 |
| 一致 | 前後一致 | 「都用『帳號』不用『帳戶編號』」 |
| 可驗證 | 可被測試 | 「限額 = 50,000」 |
| 有追蹤 | 可追溯需求 | 「對應需求 FR-001」 |

### 問題追蹤表

| 編號 | 問題 | 提問者 | 提問日期 | 回覆者 | 回覆日期 | 狀態 |
|------|------|--------|----------|--------|----------|------|
| Q001 | 手機格式驗證規則？ | 開發-A | 04/01 | BA | 04/02 | 已回覆 |
| Q002 | 限額是否含手續費？ | 開發-B | 04/02 | BA | 04/03 | 已回覆 |
| Q003 | 跨幣別如何處理？ | 開發-A | 04/03 | BA | - | 待回覆 |

### 決策記錄

```markdown
## 決策記錄

### DR-001：轉帳限額計算方式
- **日期**：2026/04/01
- **參與者**：BA, 開發 Lead, 業務代表
- **問題**：每日限額是否包含手續費？
- **決策**：每日限額只計算轉帳金額，不含手續費
- **理由**：手續費由客戶另外支付，不影響轉帳限額
- **影響**：需修改限額檢核邏輯

### DR-002：OTP 驗證門檻
- **日期**：2026/04/02
- **參與者**：BA, 資安團隊
- **問題**：超過多少金額需要 OTP？
- **決策**：單筆超過 30,000 元需 OTP 驗證
- **理由**：平衡安全性和便利性
- **影響**：需實作 OTP 驗證機制
```

## BA 實務技巧

### 溝通檢查清單

```
會議前：
□ 議程和資料已發送（至少 1 天前）
□ 相關人員已邀請
□ 設備已測試（投影、視訊）
□ 問題清單已預先整理

會議中：
□ 準時開始
□ 重點已記錄
□ 決議已確認
□ 行動項目已指派
□ 準時結束

會議後：
□ 會議記錄 24 小時內發送
□ 行動項目已追蹤
□ 相關文件已更新
```

### 文件審核檢核

```
需求規格書審核：
□ 業務背景清楚嗎？
□ 功能描述完整嗎？
□ 業務規則明確嗎？
□ 例外處理涵蓋嗎？
□ 可測試嗎？
□ 技術可行性評估了嗎？

技術設計書審核：
□ 符合需求規格嗎？
□ 技術方案合理嗎？
□ 效能考量足夠嗎？
□ 安全措施完整嗎？
□ 錯誤處理完善嗎？
```
