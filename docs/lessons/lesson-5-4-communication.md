# Lesson 5-4：與開發人員溝通技巧

## 學習目標

- 掌握與開發人員溝通的要領
- 能夠有效傳達需求
- 建立良好的協作關係

---

## 為什麼溝通很重要？

銀行 IT 專案中：
- BA 和開發人員使用不同的語言
- 誤解可能導致重工
- 良好的溝通提升效率

---

## BA 與開發的語言差異

### 業務語言 vs 技術語言

| BA 說的 | 開發聽到的 | 應該怎麼說 |
|---------|------------|------------|
| 「系統要快」 | 「要多快？」 | 「回應時間 < 3 秒」 |
| 「要安全」 | 「安全有很多種」 | 「需要 OTP 驗證」 |
| 「操作簡單」 | 「簡單的定義是？」 | 「三步驟內完成」 |

---

## 有效溝通技巧

### 1. 具體化

```
❌ 模糊：「客戶體驗要好」
✅ 具體：「頁面載入 < 2 秒，操作步驟 < 3 步」
```

### 2. 結構化

```markdown
## 需求描述
客戶需要快速轉帳功能

## 業務規則
1. 單筆限額 50,000
2. 每日限額 200,000
3. 需要 OTP 驗證

## 例外處理
- 餘額不足：顯示錯誤訊息
- 超過限額：提示限額資訊
```

### 3. 圖像化

```
┌─────────┐     ┌─────────┐     ┌─────────┐
│ 客戶    │ ──▶ │ 系統    │ ──▶ │ 核心系統 │
│ 輸入    │     │ 驗證    │     │ 執行    │
└─────────┘     └─────────┘     └─────────┘
```

## 常見溝通場景

### 需求說明會

```
準備事項：
□ 需求背景說明
□ 功能清單
□ 流程圖
□ 資料結構
□ 疑問預先整理

會議進行：
□ 逐項說明需求
□ 確認開發理解
□ 記錄技術問題
□ 約定後續事項
```

### 技術問題討論

```
BA 角度：
- 說明業務目的
- 提供業務情境
- 確認解決方案符合需求

開發角度：
- 說明技術限制
- 提供替代方案
- 評估開發成本
```

### 缺陷討論

```
BA 提供：
- 重現步驟
- 預期結果
- 實際結果
- 測試數據

開發提供：
- 問題原因
- 修正方案
- 影響範圍
```

## 文件化最佳實務

### 規格書撰寫要點

| 要點 | 說明 | 範例 |
|------|------|------|
| 清楚 | 一句話一個意思 | 「單筆限額 50,000 元」 |
| 完整 | 包含所有必要資訊 | 「含正常和例外處理」 |
| 一致 | 前後一致 | 「都用『帳號』不用『帳戶編號』」 |
| 可驗證 | 可被測試 | 「限額 = 50,000」 |

### 疑問追蹤表

| 編號 | 問題 | 提問者 | 回覆者 | 狀態 |
|------|------|--------|--------|------|
| Q001 | 手機格式驗證規則？ | BA | 開發 | 已回覆 |
| Q002 | 限額是否含手續費？ | 開發 | BA | 待回覆 |

---

## BA 實務技巧

### 溝通檢查清單

```
會議前：
□ 議程和資料已發送
□ 相關人員已邀請
□ 設備已測試

會議中：
□ 重點已記錄
□ 決議已確認
□ 行動項目已指派

會議後：
□ 會議記錄已發送
□ 行動項目已追蹤
```

### 常見溝通障礙

| 障礙 | 原因 | 解決方式 |
|------|------|----------|
| 術語不同 | 專業領域差異 | 建立詞彙表 |
| 假設不一致 | 隱含假設 | 明確說明假設 |
| 優先級不同 | 立場不同 | 協調會議 |

## 練習題

### 題目 1
將以下模糊需求改寫為具體可驗證的需求：
> 「轉帳功能要安全」

### 題目 2
設計一個需求說明會的議程。

### 題目 3
說明 BA 在缺陷討論中應該提供哪些資訊。

---

## 重點回顧

| 技巧 | 說明 |
|------|------|
| 具體化 | 用具體數字取代模糊描述 |
| 結構化 | 用結構化格式呈現需求 |
| 圖像化 | 用流程圖、架構圖說明 |
| 文件化 | 完整記錄需求和決議 |
