# Lesson 1-2：Batch vs Online 處理模式

## 1. 今日主題

深入理解 Batch（批次）與 Online（聯機）兩種處理模式的差異，以及它們在銀行系統中的應用場景。

---

## 2. 為什麼 BA 需要懂

這是 **需求分析的核心判斷**：

- 業務說「即時」，但系統可能是批次處理
- 業務說「每天」，可能是批次，也可能是有排程的 Online
- 修改批次程式，需要考慮排程、上下游依賴
- 修改 Online 程式，需要考慮回應時間、併發處理

---

## 3. 核心概念白話解釋

### Online（聯機）= 櫃台服務

想像你去銀行櫃台：

1. 你走到櫃台 → 提出請求
2. 櫃員當場處理 → 立刻給你結果
3. 如果有問題 → 當場告訴你

**特點：** 即時、一對一、馬上有結果

### Batch（批次）= 後台作業

想像銀行打烊後的後台：

1. 收集今天所有交易單據
2. 批量處理：結算、計息、對帳
3. 第二天早上看結果

**特點：** 非即時、大量處理、事後看結果

---

## 4. 正式概念與術語

| 特性 | Online（聯機） | Batch（批次） |
|------|---------------|--------------|
| 執行時機 | 使用者觸發時立即執行 | 排程時間到才執行 |
| 處理方式 | 一筆一筆處理 | 整批一起處理 |
| 回應時間 | 秒級（通常 < 3 秒） | 分鐘到小時級 |
| 使用者互動 | 有互動（輸入 → 確認 → 結果） | 無互動（背景執行） |
| 典型工具 | CICS、Web Service | JCL、Control-M |
| 適合場景 | 查詢、交易授權、資料維護 | 結算、計息、報表、對帳 |

---

## 5. 銀行業例子

### CLM（企業資金池管理）的處理分類

| 功能 | 處理方式 | 原因 |
|------|---------|------|
| 查詢資金池餘額 | Online | 客戶需要即時看到 |
| 設定歸集規則 | Online | 客戶操作後要即時生效 |
| 當日交易明細查詢 | Online | 客戶需要即時查看 |
| **每日計息** | Batch | 需要等日終，讀取全天交易後計算 |
| **日終歸集** | Batch | 需要等營業時間結束後執行 |
| **計息報表** | Batch | 批次產生，供會計核對 |
| **對帳檔案** | Batch | 批次產生，送給下游系統 |

### 為什麼計息用 Batch？

1. **需要完整資料**：計息要讀取當天所有交易，Online 時不知道「今天結束」
2. **計算複雜**：可能涉及多種利率、階梯利率、最低餘額等，處理時間長
3. **一次過帳**：所有帳戶計算完畢後，統一過帳到總帳
4. **可重跑**：如果發現問題，可以重新執行批次

## 6. JCL 範例：批次處理的啟動方式

Batch 程式需要透過 **JCL（Job Control Language）** 來執行。以下是簡化範例：

```jcl
//CLMINT01 JOB (ACCOUNTING),'CLM DAILY INTEREST',
//             CLASS=A,MSGCLASS=X,NOTIFY=&SYSUID
//*
//********************************************************************
//* CLM 每日計息批次
//* 執行時間：每日 23:00
//* 功能：計算所有資金池帳戶的利息
//********************************************************************
//*
//STEP1    EXEC PGM=CLMINT01
//INPUT    DD DSN=CLM.DAILY.TRANS,DISP=SHR
//MASTER   DD DSN=CLM.ACCOUNT.MASTER,DISP=SHR
//OUTPUT   DD DSN=CLM.INTEREST.DAILY(+1),DISP=(NEW,CATLG)
//REPORT   DD SYSOUT=*
//SYSIN    DD *
  PROCESS-DATE=2026-04-03
  INTEREST-RATE-TYPE=POOL
/*
```

### 逐行解釋

| 行 | 說明 |
|----|------|
| `//CLMINT01 JOB` | 定義一個 Job，名稱是 CLMINT01 |
| `CLASS=A` | Job 的執行優先等級 |
| `//STEP1 EXEC PGM=CLMINT01` | 執行一個 Step，程式名稱是 CLMINT01（COBOL 編譯後的程式） |
| `//INPUT DD DSN=...` | 定義輸入檔案：當日交易檔 |
| `//MASTER DD DSN=...` | 定義主檔：帳戶主檔 |
| `//OUTPUT DD DSN=...` | 定義輸出檔案：計息結果（+1 表示 GDG 新版本） |
| `//SYSIN DD *` | 程式的參數輸入 |
| `PROCESS-DATE=...` | 處理日期參數 |

## 7. BA 溝通重點

### 收到需求時，先判斷處理方式

```
業務需求：「客戶希望能即時看到今日累計利息」

BA 應該問：
1. 「即時看到」是指每次交易後更新，還是每隔一段時間更新？
2. 目前計息是日終批次，如果要在營業時間看到，需要新增一個預估計息邏輯
3. 這個預估利息要不要顯示「預估」字樣？要不要寫入正式紀錄？
```

### 確認批次相關問題

```
如果需求涉及批次：

1. 「這個功能要新增一支批次程式，還是修改現有的？」
2. 「要排在什麼時間執行？前面有哪些 Job 必須先跑完？」
3. 「執行失敗要怎麼處理？要通知誰？」
4. 「要不要產生報表或錯誤檔？」
```

---

## 8. 常見誤解

| 誤解 | 正確理解 |
|------|----------|
| Online 比較重要 | 兩者同等重要。銀行核心帳務大多靠批次完成 |
| 批次就是慢 | 批次是「非即時」，不是慢。一支批次可能幾分鐘處理幾百萬筆 |
| Online 可以取代批次 | 不行。計息、對帳、結算這些「結算型」工作，本質上就不適合即時處理 |
| 批次失敗可以馬上重跑 | 要看情況。有些批次有依賴關係，要確認上游狀態 |

## 9. 小練習

### 題目

請判斷以下需求應該用 Online 還是 Batch 處理，並說明原因：

1. **客戶在手機銀行查詢資金池餘額**
   
2. **每天凌晨產生「帳戶餘額異動報表」給會計部門**
   
3. **客戶修改資金池的歸集比例設定**
   
4. **每月 1 號計算上月利息並過帳到各帳戶**
   
5. **櫃員查詢客戶的 KYC 資料**

---

## 10. 練習解答

<details>
<summary>點擊展開解答</summary>

### 1. 查詢資金池餘額 → Online
- **原因**：客戶需要即時看到，且只是讀取資料，不涉及複雜計算

### 2. 產生餘額異動報表 → Batch
- **原因**：
  - 需要等當天所有交易結束後才能產生
  - 報表可能涉及大量資料彙整
  - 不需要即時互動

### 3. 修改歸集比例設定 → Online
- **原因**：
  - 客戶操作後期望即時生效
  - 是單一交易，不需要等批量資料

### 4. 每月計息過帳 → Batch
- **原因**：
  - 需要讀取整個月的交易資料
  - 涉及所有帳戶的大量計算
  - 需要統一過帳時間點
  - 失敗後需要可重跑

### 5. 櫃員查詢 KYC 資料 → Online
- **原因**：
  - 櫃員在服務客戶時需要即時查詢
  - 只是讀取，不涉及大量計算

</details>

## 11. 延伸思考

### 如果業務堅持要「即時計息」？

這是一個常見的需求衝突。BA 應該這樣處理：

1. **釐清需求本質**
   - 客戶要的是「即時計算並過帳」，還是「即時看到預估值」？
   
2. **說明技術限制**
   - 真正的計息需要完整交易資料，營業時間內無法確定
   - 可以提供「預估利息」功能，但要標明是預估

3. **提出替代方案**
   - **方案 A**：Online 顯示「預估利息（即時計算）」，日終批次產生「正式利息」
   - **方案 B**：每隔 2 小時批次更新一次「累計利息」欄位
   - **方案 C**：提供試算工具，讓客戶自己輸入條件計算

4. **確認業務決策**
   - 請業務決定哪個方案符合需求
   - 記錄決策過程，作為需求文件的一部分
