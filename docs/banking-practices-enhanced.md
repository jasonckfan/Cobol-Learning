## 二、存款系統 (DAS) 實務案例

### 2.1 存款業務的 Online 處理

#### 案例：活期存款提款交易

**業務流程：**
```
客戶在 ATM 提款 $5,000
    ↓
[Online] ATM 發送交易請求到 DAS
    ↓
[Online] DAS 驗證帳戶狀態、餘額、密碼
    ↓
[Online] 即時更新帳戶餘額 (餘額 - 5,000)
    ↓
[Online] 寫入交易日誌 (Transaction Log)
    ↓
[Online] 回傳結果給 ATM (成功/失敗)
    ↓
[Batch - 日終] 統計當日交易，產生報表
```

**為什麼餘額更新必須是 Online？**
- 客戶需要立即知道提款是否成功
- 必須防止超提 (Overdraft)
- 帳戶餘額是「即時狀態」，不能等批次

#### CICS 交易程式範例

```cobol
       IDENTIFICATION DIVISION.
       PROGRAM-ID. DASWD001.
      *-----------------------------------------------------------
      * DAS 提款交易程式
      * 功能：處理活期存款提款
      * 輸入：帳號、提款金額、密碼
      * 輸出：交易結果、新餘額
      *-----------------------------------------------------------
       ENVIRONMENT DIVISION.
       DATA DIVISION.
       WORKING-STORAGE SECTION.
       01  WS-ACCOUNT-MASTER.
           05  WS-ACCT-NO          PIC X(12).
           05  WS-ACCT-STATUS      PIC X(01).
               88  ACCT-ACTIVE     VALUE 'A'.
               88  ACCT-FROZEN     VALUE 'F'.
               88  ACCT-CLOSED     VALUE 'C'.
           05  WS-CURRENT-BAL      PIC S9(13)V99 COMP-3.
           05  WS-AVAILABLE-BAL    PIC S9(13)V99 COMP-3.
       
       01  WS-TRANSACTION-LOG.
           05  WS-TX-TYPE          PIC X(02) VALUE 'WD'.
           05  WS-TX-AMOUNT        PIC S9(13)V99 COMP-3.
           05  WS-TX-TIMESTAMP     PIC X(26).
       
       01  WS-RESPONSE-CODE       PIC X(02).
           88  RESP-SUCCESS       VALUE '00'.
           88  RESP-INSUFFICIENT  VALUE '51'.
           88  RESP-ACCT-FROZEN   VALUE '52'.
           88  RESP-INVALID-PWD   VALUE '53'.
       
       PROCEDURE DIVISION.
       0000-MAIN.
           PERFORM 1000-VALIDATE-ACCOUNT
           IF RESP-SUCCESS
               PERFORM 2000-CHECK-BALANCE
               IF RESP-SUCCESS
                   PERFORM 3000-UPDATE-BALANCE
                   PERFORM 4000-WRITE-LOG
               END-IF
           END-IF
           PERFORM 5000-SEND-RESPONSE
           GOBACK.
       
       1000-VALIDATE-ACCOUNT.
      *    讀取帳戶主檔，檢查狀態
           EXEC CICS READ
               FILE('ACCTMSTR')
               INTO(WS-ACCOUNT-MASTER)
               RIDFLD(ACCT-NO-INPUT)
               RESP(WS-CICS-RESP)
           END-EXEC
           
           IF ACCT-FROZEN
               MOVE '52' TO WS-RESPONSE-CODE
           ELSE IF ACCT-CLOSED
               MOVE '54' TO WS-RESPONSE-CODE
           ELSE
               MOVE '00' TO WS-RESPONSE-CODE
           END-IF.
       
       2000-CHECK-BALANCE.
      *    檢查可用餘額是否足夠
           IF WITHDRAW-AMOUNT > WS-AVAILABLE-BAL
               MOVE '51' TO WS-RESPONSE-CODE
           END-IF.
       
       3000-UPDATE-BALANCE.
      *    更新帳戶餘額
           SUBTRACT WITHDRAW-AMOUNT FROM WS-CURRENT-BAL
           SUBTRACT WITHDRAW-AMOUNT FROM WS-AVAILABLE-BAL
           
           EXEC CICS REWRITE
               FILE('ACCTMSTR')
               FROM(WS-ACCOUNT-MASTER)
               RESP(WS-CICS-RESP)
           END-EXEC.
       
       4000-WRITE-LOG.
      *    寫入交易日誌
           MOVE WITHDRAW-AMOUNT TO WS-TX-AMOUNT
           MOVE CURRENT-TIMESTAMP TO WS-TX-TIMESTAMP
           
           EXEC CICS WRITE
               FILE('TXNLOG')
               FROM(WS-TRANSACTION-LOG)
               RESP(WS-CICS-RESP)
           END-EXEC.
```

### 2.2 存款業務的 Batch 處理

#### 案例：每日利息計算

**業務規則：**
- 活期存款按日計息，按月結息
- 利率依帳戶類型不同（一般戶、VIP 戶、企業戶）
- 計息基準：每日日終餘額 × 日利率

**批次流程：**
```
[批次開始] 每日 23:00
    ↓
[Step 1] 讀取所有活存帳戶主檔
    ↓
[Step 2] 讀取當日交易明細，計算日終餘額
    ↓
[Step 3] 依帳戶類型取得適用利率
    ↓
[Step 4] 計算當日利息 (餘額 × 日利率)
    ↓
[Step 5] 累計到「應付利息」欄位
    ↓
[Step 6] 產生利息計算明細報表
    ↓
[批次結束]
```

**為什麼利息計算必須是 Batch？**
1. **資料完整性**：需要當天所有交易完成後的日終餘額
2. **計算量龐大**：可能涉及數百萬個帳戶
3. **統一時點**：所有帳戶必須在同一時間點計算
4. **可重跑性**：若發現錯誤，可以重新執行

#### JCL 批次作業範例

```jcl
//DASINT01 JOB (DAS,ACCT),'DAILY INTEREST CALC',
//             CLASS=B,MSGCLASS=X,NOTIFY=&SYSUID

//********************************************************************
//*
//JOBLIB   DD DSN=DAS.PROD.LOADLIB,DISP=SHR
//*

//* STEP 1: 排序當日交易檔，依帳號彙總
//***************************************************************
//STEP010  EXEC PGM=SORT
//SORTIN   DD DSN=DAS.DAILY.TRANS,DISP=SHR
//SORTOUT  DD DSN=&&SORTED,DISP=(NEW,PASS),
//             SPACE=(CYL,(10,5),RLSE)

//SYSIN    DD *
  SORT FIELDS=(1,12,CH,A)

//*

//* STEP 2: 計算日終餘額與利息
```

## 三、放款系統 (LAS) 實務案例

### 3.1 放款業務的 Online 處理

#### 案例：貸款還款交易

**業務流程：**
```
客戶在網銀進行貸款還款 $10,000
    ↓
[Online] 驗證還款帳戶餘額是否足夠
    ↓
[Online] 從還款帳戶扣款 $10,000
    ↓
[Online] 更新貸款帳戶本金餘額
    ↓
[Online] 計算並收取應付利息
    ↓
[Online] 寫入還款交易記錄
    ↓
[Online] 回傳還款成功訊息
    ↓
[Batch - 日終] 更新總帳、產生還款明細報表
```

**關鍵考量：**
- 還款金額分配順序：逾期利息 → 當期利息 → 本金
- 必須即時更新本金餘額，影響下次利息計算
- 若還款帳戶餘額不足，整筆交易必須 Rollback

### 3.2 放款業務的 Batch 處理

#### 案例：每月自動扣款 (Auto Debit)

**業務規則：**
- 每月固定日期從客戶帳戶自動扣款
- 扣款順序：利息 → 本金
- 若餘額不足，記錄為「扣款失敗」，產生通知

**批次流程：**
```
[批次開始] 每月 1 日 凌晨 02:00
    ↓
[Step 1] 讀取當月應扣款帳戶清單
    ↓
[Step 2] 對每個帳戶：
    ├─ 計算當期應付金額 (利息 + 本金)
    ├─ 檢查還款帳戶餘額
    ├─ 餘額足夠：執行扣款、更新貸款餘額
    └─ 餘額不足：記錄失敗、發送通知
    ↓
[Step 3] 產生扣款結果報表
    ↓
[Step 4] 發送失敗通知給客戶
    ↓
[批次結束]
```

## 四、CLM 資金池管理系統實務

### 4.1 CLM 核心功能

CLM (Cash Liquidity Management) 是企業現金管理系統，主要功能：

| 功能 | 說明 | 處理模式 |
|-----|------|---------|
| **資金歸集** | 自動將子帳戶餘額歸集到母帳戶 | Batch |
| **餘額查詢** | 查詢資金池整體餘額 | Online |
| **歸集規則設定** | 設定歸集時間、金額、條件 | Online |
| **利息分配** | 計算並分配資金池利息給子帳戶 | Batch |
| **資金調撥** | 母帳戶撥款到子帳戶 | Online/Batch |

### 4.2 資金歸集批次範例

**業務規則：**
- 每日日終自動執行
- 歸集方式：
  - **全額歸集**：子帳戶全部餘額歸集到母帳戶
  - **保留歸集**：保留一定金額，超過部分歸集
  - **定額歸集**：固定金額歸集

**批次流程：**
```
[批次開始] 每日 21:00
    ↓
[Step 1] 讀取所有資金池設定檔
    ↓
[Step 2] 對每個資金池：
    ├─ 讀取母帳戶與所有子帳戶餘額
    ├─ 依歸集規則計算各子帳戶應歸集金額
    ├─ 執行歸集：子帳戶扣款 → 母帳戶入款
    └─ 寫入歸集交易記錄
    ↓
[Step 3] 產生歸集明細報表
    ↓
[Step 4] 發送歸集結果通知給企業客戶
    ↓
[批次結束]
```

### 4.3 利息分配批次範例

**業務規則：**
- 母帳戶統一計息，享受較高利率
- 利息依各子帳戶「貢獻度」分配
- 貢獻度 = 子帳戶日終餘額 × 天數

**計算範例：**
```
資金池總利息：$10,000

子帳戶 A：平均餘額 $500,000 × 30 天 = 15,000,000 貢獻度
子帳戶 B：平均餘額 $300,000 × 30 天 =  9,000,000 貢獻度
子帳戶 C：平均餘額 $200,000 × 30 天 =  6,000,000 貢獻度
─────────────────────────────────────────────────
總貢獻度：30,000,000

分配結果：
子帳戶 A：10,000 × (15,000,000 / 30,000,000) = $5,000
子帳戶 B：10,000 × ( 9,000,000 / 30,000,000) = $3,000
子帳戶 C：10,000 × ( 6,000,000 / 30,000,000) = $2,000
```

### 5.2 批次相依性管理

**關鍵原則：** 下游批次必須等待上游批次成功完成

```
DAS 交易彙總批次 (DASTXN01)
            │
            ▼
    DAS 利息計算批次 (DASINT01)
            │
            ▼
    總帳更新批次 (GLPOST01)
            │
            ▼
    對帳單產生批次 (STMTGEN01)
```

**Control-M 排程工具設定範例：**
```
Job: DASINT01
  - 排程時間：每日 23:00
  - 前置條件：DASTXN01 必須完成 (RC=0)
  - 逾時警告：超過 2 小時未完成則發送告警
  - 失敗處理：重試 3 次，仍失敗則通知值班人員
```

### 5.3 批次異常處理

#### 常見異常情況

| 異常類型 | 可能原因 | 處理方式 |
|---------|---------|---------|
| **ABEND (異常終止)** | 程式邏輯錯誤、資料異常 | 查看 SYSOUT、修正後重跑 |
| **逾時 (Timeout)** | 資料量過大、資源競爭 | 調整排程時間、優化程式 |
| **檔案不存在** | 上游批次未完成 | 確認前置批次狀態 |
| **資料不一致** | 交易檔與主檔不符 | 執行資料修復程式 |

#### 批次重跑流程

```
批次失敗
    ↓
[Step 1] 查看 SYSOUT 確認錯誤原因
    ↓
[Step 2] 修正問題（程式錯誤、資料錯誤）
    ↓
[Step 3] 決定重跑方式：
    ├─ 從失敗 Step 重跑 (RESTART)
    ├─ 從頭重跑整個 Job (RERUN)
    └─ 手動補資料，跳過此 Job
    ↓
[Step 4] 確認重跑結果
    ↓
[Step 5] 檢查下游批次是否需要調整
```

## 六、對帳與報表生成

### 6.1 對帳流程

**日終對帳檢查項目：**

| 檢查項目 | 說明 | 處理方式 |
|---------|------|---------|
| **借貸平衡** | 總帳借方 = 貸方 | 若不平衡，停止後續批次 |
| **交易筆數** | Online 交易筆數 = Batch 處理筆數 | 核對差異 |
| **金額總和** | 交易金額總和一致 | 追蹤差異來源 |
| **帳戶餘額** | 日初餘額 + 交易 = 日終餘額 | 驗證計算正確性 |

**銀行常見報表：**

| 報表名稱 | 產生時間 | 用途 | 格式 |
|---------|---------|------|------|
| **日計表** | 每日日終 | 會計科目每日餘額 | 紙本/PDF |
| **交易明細表** | 每日日終 | 各帳戶交易明細 | 電子檔 |
| **資產負債表** | 每月月底 | 財務報表 | 紙本 |
| **利息明細表** | 每月結息日 | 各帳戶利息明細 | 電子檔 |
| **對帳單** | 每月月底 | 寄送客戶 | 紙本/Email |

## 七、BA 溝通實務

### 7.1 需求分析時的關鍵問題

#### 判斷 Online vs Batch

```
當業務說「即時」時，問清楚：
├─ 「即時」是指秒級回應，還是幾分鐘內？
├─ 這個資料現在存在嗎？還是需要等日終？
├─ 如果現在顯示「預估值」，日終顯示「正式值」，可以接受嗎？
└─ 有多少使用者會同時使用這個功能？

當業務說「每天」時，問清楚：
├─ 是固定時間執行，還是隨時可以手動執行？
├─ 執行時間有沒有限制？（例如必須在 23:00 前完成）
├─ 失敗了要怎麼辦？可以隔天補做嗎？
└─ 有沒有前置條件？（例如必須等某個系統完成）
```

### 7.2 影響分析檢查清單

**修改 Online 程式時：**
- [ ] 回應時間是否會增加？
- [ ] 併發處理是否會有競爭條件？
- [ ] 錯誤處理是否完整？
- [ ] 是否需要修改畫面規格？

**修改 Batch 程式時：**
- [ ] 執行時間是否會增加？（會不會影響下游批次？）
